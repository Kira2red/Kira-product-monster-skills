# 产品白皮书维护 Skill

## 用途
将 PRD 的增量变更合并到产品白皮书中,保持白皮书为全量功能文档。

## 触发条件
当用户要求:
- "将 PRD 合并到白皮书"
- "更新产品白皮书"
- "基于 PRD 生成白皮书"
- "白皮书增量更新"

## 核心原则
1. **白皮书按功能模块逻辑组织**,不按版本时间线
2. **尽量保留 PRD 原文**,保持描述详细程度一致
3. **白皮书文件名需体现版本号**,格式: `产品白皮书_v[版本号].md`
4. **长文档需逐模块合并**并等待用户确认
5. **合并前必须校验**功能模块是否对应,不对应则警告停止

## 执行流程

### 阶段 0: 判断模式
检查白皮书文件是否存在:
- **不存在** → 进入"初版生成模式"(阶段 1)
- **存在** → 进入"增量合并模式"(阶段 2)

### 阶段 1: 初版生成模式
当白皮书文件不存在时:

1. **读取 PRD 文档**
2. **提取核心功能模块**
3. **生成白皮书结构**:
   - 文档头部(版本、更新日期、覆盖范围)
   - 修改日志
   - 功能概述
   - 各功能模块详细内容
   - 功能依赖关系
4. **保存文件**到 `产品需求/` 目录,文件名格式: `[产品名]产品白皮书_v[版本号].md`
5. **告知用户**白皮书已生成

### 阶段 2: 增量合并模式
当白皮书文件已存在时:

#### 2.1 读取文档
- 读取现有白皮书
- 读取新版本 PRD

#### 2.2 模块对应校验
- 提取 PRD 中的功能模块列表
- 提取白皮书中的功能模块列表
- **对比模块结构**:
  - 如果 PRD 包含白皮书中不存在的模块 → 标记为"新增"
  - 如果 PRD 中的模块描述与白皮书不同 → 标记为"修改"
  - 如果白皮书中的模块在 PRD 中不存在 → 标记为"删除"(需特别注意)

#### 2.3 合并策略确认
向用户展示变更摘要:
```
检测到以下变更:
- 新增模块: [模块A, 模块B]
- 修改模块: [模块C]
- 删除模块: [模块D] ⚠️

是否继续合并?
```

#### 2.4 执行合并
根据变更类型执行不同策略:

**新增模块**:
- 在白皮书中找到合适的插入位置(按功能逻辑分组)
- 插入完整的模块内容
- 保持 PRD 原文描述

**修改模块**:
- 定位白皮书中对应的模块章节
- 精准替换修改的部分
- 保持未修改部分不变

**删除模块**:
- ⚠️ **二次确认**:"确定要删除 [模块D] 吗?此操作不可恢复"
- 用户确认后才执行删除
- 精准删除,不误删其他内容

#### 2.5 更新元数据
- 更新版本号(根据变更类型决定主版本/次版本/修订号)
- 更新"最后更新"日期
- 在"修改日志"中添加本次变更记录

#### 2.6 分段确认机制
如果 PRD 或白皮书超过 3000 字:
- 逐模块进行合并
- 每完成一个模块后暂停
- 向用户展示合并结果
- 等待用户确认后继续下一个模块

### 阶段 3: 完成
- 保存更新后的白皮书
- 向用户报告:
  - 新版本号
  - 变更摘要
  - 文件保存路径

## 注意事项

### 版本号规则
- **主版本号**(X.0.0): 重大功能重构或架构变更
- **次版本号**(x.Y.0): 新增功能模块
- **修订号**(x.y.Z): 功能细节修改或bug修复

### 安全机制
1. **模块不对应时必须警告**并停止合并
2. **删除操作需二次确认**避免误删
3. **长文档分段处理**避免一次性修改过多内容
4. **保留 PRD 原文**避免信息丢失

### 白皮书结构要求
```markdown
# [产品名]产品白皮书

**版本**: vX.Y.Z
**最后更新**: YYYY-MM-DD
**覆盖范围**: [简要说明包含哪些功能模块]

---

## 修改日志

### vX.Y.Z (YYYY-MM-DD)
- 变更内容

---

## 1. 功能概述
[总体介绍]

---

## 2. [功能模块A]
[详细内容]

---

## N. 功能依赖关系
[模块依赖、数据同步、接口依赖]
```

## 示例对话

**用户**: "将这个 PRD 合并到充电白皮书中"

**助手**:
1. 读取白皮书和 PRD
2. 检测到新增3个模块、修改1个模块
3. 展示变更摘要并询问确认
4. 用户确认后执行合并
5. 更新版本号为 v1.1.0
6. 报告完成

---

**重要**: 每次合并都要在白皮书的"修改日志"中记录变更,方便追溯历史。
