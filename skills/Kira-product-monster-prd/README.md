# 2RED Product Monster PRD

一个面向产品经理的 PRD 撰写技能，用于将口头需求、草图、HTML 原型转换为结构化、研发可实施的产品需求文档。适用于 AI 编程助手（如 Claude Code、Kilo 等）。

---

## 适用场景

- **撰写 PRD**：从零开始写一份完整的产品需求文档
- **扩写 PRD**：将简陋的需求描述扩充为细节完备的规范文档
- **重写 PRD**：将套娃式、散装的旧 PRD 重构为共享层 + 模块层的标准结构
- **评审 PRD**：检查 PRD 类型是否明确、收益预估是否量化、版本管理是否完整、验收标准是否可执行

---

## 核心能力

### 共享层与模块层分离

传统 PRD 常见"套娃"问题 —— 文档中有多个功能，每个功能下面各自嵌套一套"项目背景 → 业务流程 → 详细说明"。这会导致重复劳动、版本信息碎片化，以及研发和测试无法一眼看清整体。

本技能强制遵循两层分离：

| 层级 | 包含内容 | 特点 |
|------|---------|------|
| **共享层** | PRD 类型、背景与收益、业务流程、版本管理、系统关联、验收标准、辅助材料 | 整份文档仅一套，所有功能共用 |
| **模块层** | 各功能的详细交互、状态流转、异常处理 | 每个功能一个子章节，互不重复 |

如果一份 PRD 包含 6 个功能：共享层写一次，6 个子章节只写各自的位置、交互、状态、边界。

### 微观交互规范

每个功能模块下强制覆盖 8 个维度的关键要素：

| 维度 | 覆盖内容 |
|------|---------|
| 文本与数据展示 | 静态/动态内容、超长截断、空数据 |
| 按钮与操作 | 可用/禁用条件、点击反馈、防抖、成功/失败 |
| 表单与输入 | 字符限制、校验时机、错误提示 |
| 加载态 | 首屏骨架屏、组件加载、加载失败重试 |
| 弹窗 | 触发/关闭方式、遮罩层交互、多弹窗优先级 |
| 错误态 | 接口报错提示 + 重试、权限不足引导 |
| 异常与降级 | 网络超时、关键功能失效兜底、离线可用性 |
| 边界数值 | 刷新频率、超时阈值、分页大小、文件上限、操作频次 |

### 流程与状态图表

当状态节点 ≥ 3 个、涉及多方交互或 ≥ 2 个条件分支时，自动输出：
- **PlantUML**：适用于时序图、泳道图（多角色协同）
- **Mermaid**：适用于状态机图、业务流程图

图表文件随 PRD 一并保存，PRD 中使用相对路径引用。

## PRD 文档结构

```
PRD 文档名.md
├── §0  PRD 类型判定（功能型 / 策略型 / 修复型 / 架构型）
├── §1  项目背景与收益预估（金字塔：用户收益 → 业务收益 → 商业收益 → 不做风险）
├── §2  业务流程简述
├── §3  版本管理（PRD 修订记录 + 产品版本信息）
├── §4  系统关联与依赖（模块数据流、接口依赖、耦合度评估）
├── §5  详细功能说明（每个功能一个子章节）
│    ├── 5.1  功能一
│    │    ├── 位置/目标
│    │    ├── 交互逻辑与状态流转
│    │    ├── 加载态
│    │    ├── 错误态
│    │    └── 边界数值
│    ├── 5.2  功能二
│    └── ...
├── §6  流程与状态图表（PlantUML / Mermaid）
├── §7  数据埋点与实验设计（策略型 PRD 必须）
├── §8  验收标准（四要素：场景/前置条件/操作/期望结果/判定）
└── §9  辅助材料清单（UI 设计稿、音频 Demo、流程图、线框图）
```

## 使用方式

### 安装

将 `2red-product-monster-prd/` 文件夹复制到 AI 编程助手的 skills 目录中。以 Kilo 为例：

```
.kilo/skills/2red-product-monster-prd/
├── README.md
└── SKILL.md
```

### 触发

安装后，当你说出以下指令时，助手会自动调用该技能：

- "帮我写一份 PRD，功能是..."
- "把这段需求描述扩充成 PRD"
- "重写这份旧 PRD"
- "评审下这份 PRD"

也可以在助手中显式指定：`@2red-product-monster-prd 写一份乘客车控的 PRD`

### 渐进式编写

对于超过 3 个模块的复杂 PRD，技能会采用渐进式流程：
1. 先输出共享层 + 第一个功能模块，让你确认方向
2. 后续每个功能追加前，先检查共享层是否需要修订（如收益覆盖、依赖新增）
3. 全部功能完成后，统一修正编号和交叉引用

## 写作铁律

1. **严格序号层级**：必须遵循 `1. → 1.1 → 1.1.1 → •` 的嵌套关系，不允许层级混乱
2. **纯中文表达**：除行业标准专有名词（ID、API、CSV、TTS 等）外，一律使用中文
3. **业务与交互视角**：只定义业务规则、状态机、数据流向和前端表现，不涉及研发实现细节

---

## 辅助材料

每个 PRD 应按需附上：
- UI 截图/设计稿（涉及 UI 变更时）
- 音频 Demo（涉及语音交互时）
- 流程图（分支 > 3、模块 > 3、步骤 > 3 时）
- 线框图（涉及新页面时）
